Essential Business Concepts

When working in the main planning view, there are some basic constructs and conventions the user should be aware of in order to get the most out of the available functionality:

Aggregation Levels

In order to provide a more fluid user experience, the data selection parameters include product and channel aggregation levels. Data for the selected portions of the product and site hierarchies will be pre-aggregated to these levels in temporary database tables, thus reducing the total number of rows that must be read/written and improving overall system performance.

Aggregation levels can be changed at any time without consequence to the work users have done. Data will simply be re-aggregated/disaggregated. Note, however, that actual values are always stored at the lowest plan levels in the database, regardless of what levels you are currently aggregated to. Changes made at higher levels are simply broken down by percent contribution of the previous values.

  • Example: based on the aggregation level selected, the user will see values for that level as well as those above. In the chart below, if the user selects Total Company as the Channel Agg Level, the system will only display a single section for Vandelay Industries and $22,500.00 in sales, but if the user selected Channel as the Channel Agg Level, the system would display separate sub-sections for Retail, Outlet, and Ecomm, within the country sections for US and UK and also a grand total section for Vandelay Industries.

PivotGrid Functionality

Very similar to the popular Microsoft Excel Pivot Table but with enhanced functionality, the PivotGrid allows users to quickly aggregate, disaggregate, and rearrange the view as well as make edits to all plannable metrics at multiple levels.

By clicking the arrows at the left corner of major sections, such as the Brand section Great Value in the example above, users can work at the total Brand level for both Channels shown:

Now, any changes made will be for both Retail and Outlet Channels of the Great Value Brand. Changes made at this level will be broken down to the individual channels in the database, based on the percent contribution of the existing values of each. Thus, if the user expands the section again, the Retail and Outlet Channels will be updated accordingly.

Similarly, horizontal sections, such as the month and year sections shown above, can be collapsed by clicking the arrows at the left of each to work at aggregate levels of time:

In the first example, changes made will now affect the entire month, and in the second, changes will affect the entire year, and all changes made will be broken down to the lower levels via percent contribution of the existing values.

Finally, fields in the PivotGrid can also be dragged and dropped to other sections to quickly rearrange the view:

Dragging the ProductLevelDescription field at the left up to the column area of the grid will rearrange the grid so all Brands can be viewed side-by-side for each week.

Similarly, fields can be dragged from the column area to the row area. In the above example, dragging the Month and Week fields to the row area allows the user to view time vertically while still having all Brands arranged side-by-side for easy comparisons.

Fields Customization

The fields visible to the user in a PivotGrid can be customized by clicking the button in the view header area to launch the Fields Customization dialog:

In the Simple tab, users can simply drag fields from the various areas of the PivotGrid into others. The order the fields appear in each section defines the order they appear in the actual grid. Order can also be manually defined by typing over the numbers in the column to the right of each field name.

In the Advanced tab, the Pivot Area of each field can be selected via dropdown menu, or, again, set manually by overwriting the order Index number. The real benefit to the Advanced tab, however is in the ability to use the Multiple Changes function to move many fields to a desired Pivot Area in a single action. See Multiple Changes below for more information.

Saving/Restoring Layouts

Changes made to the screen layout through use of the Fields button and/or PivotGrid manipulation can be saved and quickly recalled later:

  • Saving: right clicking in the pivot field in the row area nearest the Data field (Week in the example below) or anywhere in the data grid area will bring up a context menu:


    Click Save Layout to launch the Save Layout dialog:

    Enter a Name for the layout, an optional Description, and a privacy option – Private, Public, or Shared (if Shared, click to define the users with which to share the layout). If you wish to make the layout you are saving the Default system layout for the current view, check Default. Click OK and then the layout will be saved for future recall

  • Restoring: right click as shown in Saving, above, to bring up the context menu:

Click Restore Layout then choose the layout you wish to recall to switch the view to the chosen layout

Multiple Changes

Users can make changes to multiple cells in a single action by selecting the desired cells and right clicking to bring up the context menu:

Click Multiple Changes to launch the Multi Data Change Form dialog:

Here, users can choose a number of different actions by which to manipulate the selected cells:

  • New Value: sets a new value with which to overwrite the values of the selected cells
  • Increase/Decrease: increases/decreases the selected cells by the specified amount or percent
  • Split Value: splits the entered value across the selected cells either as a weighted split based on the current cell values or evenly
  • Copy From: allows the user to choose a number of cells equal to the number currently selected to copy values from; click the Select Fields button, select the desired cells in the view, then click OK, and review the value mappings in the Results Preview section
  • Round new value: selects a rounding method for the new values – up, down, or nearest
  • Note: only applies when new values have decimals

  • When alternate (non-default) calc toggles are selected, they appear in darker gray for easy identification at a glance
  • The Calc Toggles view can be collapsed by clicking the arrow at the top left adjacent the view title:

    Once collapsed, the view will be visible only as a tab at the right hand side of the screen and can be restored by clicking the arrow near the view title again:

DC Stock Inclusion

Units owned by sites of type DC are now rolled into the last actualized period of the active plan so they can be considered as part of total inventory ownership by product/channel hierarchy level. Within each product hierarchy level, available DC stock is split among all relevant site hierarchy levels based on the following system policy:

  • DC Stock Splits Future Number of Periods: this policy sets the number of periods forward the system uses to calculate the ideal splitting of available inventory among relevant site hierarchy levels; default setting is 8

The system calculates the relative need of each site hierarchy level by comparing the planned Sales Units for the number of periods specified in the DC Stock Splits Future Number of Periods policy against the planned Receipt Units for the same number of periods plus the last actualized period’s EOP Units. The system then divvies available inventory among the relevant site hierarchy levels and adds the split quantities to the EOP Units of last actualized period for each and carries them forward into the BOP Units for the following period

  • In order for DC stock splitting to occur, the DCs containing the stock you want split cannot be included in the master plan, as this would cause a duplication of stock quantities
  • Example: in this example, all DCs have been placed in a separate DC channel to contain the DC stock within a single channel level; this level is then excluded from the plan by unchecking the box next to the DC  level in the bottom portion of the Maintain Financial Plan dialog. This is an easy way to ensure the DC sites are excluded from the plan without having to alter the site hierarchy
  • Stock in DCs that are part of enabled channel levels of the plan will show as part of the stock on hand for those channels; this is recommended for cases where there is a 1-1 mapping between channels and DCs and stock from a single DC will not be shared among multiple channels
  • DC stock split quantities can be viewed in the Main Planning view in the fields Actual EOP Units Cluster (if the Inventory Is Shared policy is set to False) and Actual EOP Units DC Split for reference
  • Note: the Inventory Is Shared system policy is used only for legacy Cost based plans to share incoming inventory among the site levels and is not used in Hybrid based plans
  • DC stock is stored against the Master plan and split from there, ensuring that stock is not double-counted in branch plans beneath
  • DC stock splits are recalculated dynamically, either when daily maintenance procedures run, or when changes to Sales Units and/or Receipt Units are made to any periods specified by the system policy and Save is clicked
  • Because it is possible for multiple versions of branch plans for the same product/channel levels to exist (upside, downside, and flat scenario plans, for example), if all channels sharing the DC stock for a given product level are not present in a branch, the system will use the planned values from the channels present in the branch and the master plan values for the same periods for the channels not present in the branch to determine DC stock splits in each branch. This is because, if multiple branch versions exist, it is not possible for the system to determine which one to use in the calculation of relative need by channel.
  • Important: in the case where multiple branch plans exist for the same product level but different channel levels, it is possible that DC stock splits for those branches will recalculate when branches are merged into the Master plan, necessitating review of the receipt plans in each branch
  • Helpful Hint: in this scenario, it is recommended to have a “master branch” containing all product and channel levels so that branches can be merged into it and reviewed/adjusted at aggregate levels prior to merging into the Master plan
